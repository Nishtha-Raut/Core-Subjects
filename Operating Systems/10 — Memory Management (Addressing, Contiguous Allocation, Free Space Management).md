# 10 — Memory Management (Addressing, Contiguous Allocation, Free Space Management)

## 1. Why Memory Management?

- In a multi-programming environment, we keep **multiple processes in main memory** to maximize CPU utilization and system responsiveness.
- To realize this, we must **share** the main memory among processes — meaning the OS must **manage** memory allocation for all these different processes.

---

## 2. Logical vs Physical Address Space

### Logical Address
- An address **generated by the CPU**.
- Basically the address of an instruction or data used by a process.
- The **user can access** the logical address of their process.
- User has **indirect** access to the physical address, via the logical address.
- Logical address **doesn't exist physically** — hence aka **Virtual Address**.
- The set of all logical addresses generated by a program = **Logical Address Space**.
- **Range:** `0` to `max`.

### Physical Address
- An address loaded into the **memory-address register** of the physical memory.
- The **user can never access** the physical address directly.
- It's a real location in main memory.
- The set of all physical addresses corresponding to logical addresses = **Physical Address Space**.
- Computed by the **Memory Management Unit (MMU)**.
- **Range:** `(R + 0)` to `(R + max)`, for base value `R`.

### The MMU (Memory Management Unit)
- The **runtime mapping from virtual to physical address** is done by a **hardware device** called the MMU.
- The user's program generates the logical address and *thinks* it's running at that address — but the program actually needs physical memory to execute.

```
                    Relocation Register
                          [14000]
                             |
CPU --Logical Address(356)--> (+) --Physical Address(14356)--> Memory
```

**Real-life analogy:** Think of Logical Address like an **apartment number** ("Apt 12") in your building's internal numbering, while the Physical Address is the **GPS coordinate** the postal service actually uses. You (the process) only ever say "Apt 12" — the building management (MMU) translates that to the real-world location by adding the building's known base offset (relocation register), so mail actually arrives at the right physical spot.

### How OS Manages Isolation & Protection (Memory Mapping & Protection)

- OS provides the **Virtual Address Space (VAS)** concept to separate memory spaces.
- We need the ability to determine the **range of legal addresses** a process may access, and ensure the process can access **only** those legal addresses.
- **Relocation register**: contains the value of the smallest physical address (**Base address `R`**).
- **Limit register**: contains the range of logical addresses (e.g., relocation = 100040, limit = 74600).
- **Each logical address must be less than the limit register.**
- MMU maps the logical address dynamically by **adding** the relocation register's value.
- On a **context switch**, the dispatcher loads the relocation and limit registers with the correct values for the new process. Since every CPU-generated address is checked against these registers, this **protects both the OS and other users' programs/data** from being modified by the running process.
- Any attempt by a user-mode program to access OS memory or another process's memory results in a **trap** to the OS, treated as a fatal error.

### Address Translation Flow
```
CPU --Logical Address--> [< Limit Register?]
                              |
                        Yes   |   No
                         |         |
                         v         v
              [+ Relocation Reg]  Trap: addressing error
                         |
                         v
                  Physical Address --> Main Memory
```

**Real-life analogy:** This is like an airport security check with a **boarding pass boundary check**. Your boarding pass (logical address) is checked against the flight's valid seat range (limit register) — if your requested seat number exceeds the plane's capacity, you're rejected (trap: addressing error). If valid, it's translated to your actual physical seat location by adding the cabin's starting offset (relocation register).

## 3. Allocation Methods on Physical Memory

Two broad categories:
1. **Contiguous Allocation** — each process occupies a single continuous block of memory.
2. **Non-contiguous Allocation** — a process's memory can be split across multiple, scattered locations (covered in File 11: Paging & Segmentation).

---

## 4. Contiguous Memory Allocation

> Each process is contained in a **single contiguous block of memory**.

### 4.1 Fixed Partitioning
- Main memory is divided into **partitions of equal or different sizes**, decided **upfront**.

```
   Operating System   | partition for OS
   -------------------
   3MB  Process P1     | partition 1 (3MB used, 1MB unused)
   -------------------
   3MB  Process P2     | partition 2 (3MB used, 1MB unused)
   -------------------
   3MB  Process P3     | partition 3 (3MB used, 1MB unused)
   -------------------
   3MB  Process P4     | partition 4 (3MB used, 1MB unused)
```
- Even though the **unused 1MB spaces total 4MB**, Process P5 (needing 4MB) **can't be loaded**, because that 4MB isn't **contiguous**.

### Limitations of Fixed Partitioning

| Limitation | Description |
|---|---|
| **Internal Fragmentation** | If a process is smaller than its partition, the leftover space within that partition is **wasted** and unused. |
| **External Fragmentation** | Total unused space across various partitions **can't** be combined to load a process, even though enough total space exists — because it's not contiguous. |
| **Limitation on process size** | A process **larger** than the biggest partition can never be loaded. |
| **Low degree of multi-programming** | Since partition sizes are fixed, the number of processes that can be loaded doesn't adapt to actual process sizes. |

**Real-life analogy:** Fixed partitioning is like a **parking lot pre-painted with fixed-size parking spots** — say, all spots sized for sedans. A truck (large process) simply can't park anywhere, even if there's technically enough *total* empty space scattered across multiple sedan spots. And if a tiny smart car parks in a sedan spot, the extra space around it is wasted (internal fragmentation).

### 4.2 Dynamic Partitioning
- Partition size is **not declared upfront** — it's declared **at the time of process loading**, exactly matching the process's actual size.

```
   Operating System   | partition for OS
   -------------------
   Process P1 (5MB)    | partition 1
   -------------------
   Process P2 (2MB)    | partition 2
   -------------------
   Process P3 (3MB)    | partition 3
   -------------------
   Process P4 (4MB)    | partition 4
```

### Advantages over Fixed Partitioning
1. **No internal fragmentation** (partition = exact process size).
2. **No limit on process size** (as long as total free memory can fit it).
3. **Better degree of multi-programming**.

### Limitation: External Fragmentation
- As processes complete and leave, memory gets fragmented into small, scattered free chunks.

```
   Operating System | partition for OS
   -----------------
   5MB Free           | partition 1  (P1 completed)
   -----------------
   Process P2 (2MB)   | partition 2
   -----------------
   3MB Free           | partition 3  (P3 completed)
   -----------------
   Process P4 (4MB)   | partition 4
   -----------------
                       Process P5 (8MB) -- CANNOT be loaded!
                       Even though 5MB + 3MB = 8MB free space exists,
                       it's not contiguous.
```

**Real-life analogy:** Dynamic partitioning is like **valet parking that sizes the spot exactly to each car** as it arrives — efficient use of space initially. But as cars come and go, you end up with **scattered small gaps** between remaining cars — even if the gaps add up to enough total space for a bus, the bus can't squeeze into any single gap.

## 5. Free Space Management

### 5.1 Defragmentation / Compaction
- Dynamic partitioning suffers from external fragmentation.
- **Compaction** minimizes the probability of external fragmentation by making all free partitions **contiguous** — all loaded partitions are shifted/brought together.
- This lets bigger processes be stored, since merged free space can now satisfy larger requests. Also called **defragmentation**.
- **Drawback:** System efficiency **decreases during compaction**, since all the scattered free spaces need to be physically moved/transferred to one place — a costly operation.

**Real-life analogy:** Compaction is like a movie theater usher walking through a partially-full row and asking everyone to **shift down to one end**, consolidating all the empty seats together at the other end — so a large group can now sit together. It takes time/effort (moving people = moving memory) but frees up a usable contiguous block.

### 5.2 How is Free Space Represented?
- Free holes in memory are represented by a **free list** (a **Linked-List** data structure).

### 5.3 Allocation Algorithms — Satisfying a Request of Size `n`

| Algorithm | Strategy | Pros | Cons |
|---|---|---|---|
| **First Fit** | Allocate the **first** hole that's big enough. | Simple, easy to implement, **fast / low time complexity**. | May leave awkward-sized leftover holes. |
| **Next Fit** | Like First Fit, but always **starts searching from the last allocated hole** (not from the beginning each time). | Same advantages as First Fit, spreads search more evenly. | Similar fragmentation issues as First Fit. |
| **Best Fit** | Allocate the **smallest** hole that's big enough. | **Lesser internal fragmentation.** | May create **many tiny holes**, causing **major external fragmentation**. **Slow** — must scan the whole free list to find the smallest fit. |
| **Worst Fit** | Allocate the **largest** hole available. | Leaves **larger** leftover holes that may accommodate future processes. | **Slow** — must scan whole list. Generally performs poorly in practice. |

**Real-life analogy — imagine parking a car in one of several available empty lots of varying sizes:**
- **First Fit** = park in the **very first** lot big enough, even if it's much bigger than needed — quick decision, don't overthink it.
- **Next Fit** = same as First Fit, but you resume searching from **where you last parked**, not from the start of the street each time.
- **Best Fit** = drive around the **whole street**, find the **snuggest-fitting** lot — minimizes wasted space around your car, but you might leave many awkward tiny unusable gaps elsewhere, and it takes longer to decide.
- **Worst Fit** = deliberately park in the **biggest** available lot, leaving a large leftover space that (hopefully) can fit another car later.

## Interview Q&A — Memory Management

**Q1: Why can't a user process directly know or use physical addresses?**
> Because physical addresses are managed exclusively by the OS/MMU for isolation and security — if user processes could directly reference physical memory, one process could read/corrupt another process's or the OS's memory. Logical (virtual) addressing lets the OS enforce boundaries via the relocation and limit registers, translating safely at runtime.

**Q2: What's the practical difference between internal and external fragmentation, and which allocation scheme suffers from which?**
> **Internal fragmentation** = wasted space *within* an allocated partition (space allocated but unused) — happens in **Fixed Partitioning**. **External fragmentation** = wasted space *between* allocated partitions, scattered as small unusable free chunks — happens in **Dynamic Partitioning**. Fixed partitioning trades internal fragmentation for simplicity; dynamic partitioning trades external fragmentation for a tighter per-process fit.

**Q3: Why is Best Fit not always the best practical choice despite its name?**
> Because it tends to leave many very small, often useless leftover holes (since it always picks the tightest fit, the "leftover" from that fit is usually tiny and hard to reuse) — leading to significant external fragmentation over time, and it's also slower since it must scan the entire free list to find the true best fit, unlike First Fit which stops at the first match.

**Q4: How does compaction solve external fragmentation, and why isn't it done constantly?**
> Compaction physically moves all allocated blocks together, consolidating all free space into one large contiguous chunk. It's not done constantly because moving memory content around is **expensive** (CPU cycles, memory bandwidth) and pauses/slows down running processes during the operation — so it's typically triggered only when fragmentation becomes severe enough to block new allocations.

**Q5: How does this relate to real-world heap memory management in programming languages?**
> Languages like Java/Python/.NET use **garbage collectors** that perform compaction-like operations (e.g., a "mark-compact" or "copying" GC) to consolidate free heap space after collecting garbage objects — directly analogous to OS-level compaction, just operating within a process's heap instead of across the whole system's physical memory.
